# InnerLife v2.2 验证记录

## 目标

证明待分享念头不再只是被动存放，而能经过二次判断形成会话内分享闭环。

## 验收范围

- 会话开始时可选择一条成熟的主动开场内容。
- 对话中可结合上下文选择一条自然带入内容。
- 每次最多选择一条。
- 每日主动上限和重复冷却真实生效。
- `used`、`deferred`、`discarded` 形成完整回报。
- 延后内容继续保留，并记录延后次数。
- 过期内容自动放弃。
- 每次评估和结果都有记录。
- 旧数据库自动增加新结构。
- 不产生会话外通知。

## 自动验证

```bash
python -m pytest -q
python server/test_mcp_smoke.py
python -m innerlife.cli run-scenarios scenarios/phase0_cases.jsonl --json
```

## 结果

- 43 项自动测试全部通过。
- 9 项原有边界场景全部通过。
- MCP 真实连接检查通过，共 16 个工具。
- 旧版数据库结构自动升级测试通过。
- 成熟的 `proactive_allowed` 内容可以进入会话开始判断。
- DeepSeek 在没有自然时机时选择继续等待，没有为了主动而硬说。
- 提供相关对话上下文后，DeepSeek 选择自然带入正确内容。
- 宿主回报 `used` 后，内容从 pending 队列消失。
- `deferred` 保留内容并增加延后次数。
- 每日主动上限、重复冷却和过期放弃均通过测试。
- 分享评估和最终结果可以完整查询。

## 真实模型验收

使用临时数据库和线上 DeepSeek，创建了一条等待 24 小时的
`proactive_allowed` 内容。

新会话开始时，模型判断当前没有上下文，不应为了主动而硬说，返回 `wait`。

随后提供实际相关上下文：

> 用户正在询问为什么待分享内容一直没有主动说出来，以及怎样让主动表达更自然。

模型返回 `share_now + natural`，并选择唯一相关内容。宿主回报 `used` 后，
pending 队列变为空。

## 结论

v2.2 已经完成“想说 → 等待 → 判断时机 → 会话内表达 → 回报结果”的闭环。
系统仍然不会在会话外主动发送通知。
